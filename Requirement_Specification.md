# 施工记录智能录入与批量解析系统 - 需求规格说明书

## 1. 项目概述
### 1.1 项目背景
在建筑工程、市政工程、装修工程等需要严格进度管理的领域，施工日志（记录）是项目管理、质量追溯和资金结算的核心凭证。传统手工记录施工日志效率极低、易出错、信息滞后。当前虽然已开发了电子化的施工记录录入系统，但依旧依赖人工手动敲击录入，录入速度慢。此外，许多项目积累了大量历史手工记录（纸质扫描件、照片、PDF等），需要全量补录入存量系统。

### 1.2 建设目标
为彻底解决工程现场数据采集慢、历史数据积压的痛点，本项目拟建设“施工记录智能录入与批量解析系统”。系统将基于大语言模型（LLM）、视觉语言模型（VLM）、图像识别（OCR）与语音识别（ASR）技术，接收施工现场的语音口述、随手拍照片、历史扫描PDF，自动提取结构化业务字段，并**无缝对接至现有系统生成“草稿”**，实现“AI高并发预填 + 人工低成本复核”的极速录入闭环。

---

## 2. 核心业务痛点
1. **输入效率低下**：现场施工人员佩戴手套、环境嘈杂，手机端手动打字填表困难。
2. **历史台账积压**：过去几个月甚至几年的纸质施工日志堆积如山，人工批量录入耗时耗力，成本极高。
3. **数据格式非标**：一线人员记录的内容口语化严重、格式随意、字迹潦草，或者表格排版错乱。
4. **孤岛效应**：纯AI结构化提取缺乏业务闭环，无法直接赋能现有的项目进度管理软件。

---

## 3. 总体业务流程

1. **多端多模态采集**：
   - 现场人员通过移动端发送一段“语音”或拍摄一张“现场照片/纸质记录表”，支持弱网环境下的断点续传离线机制。
   - 内业人员在PC端打包上传几百份“历史PDF扫描件”进行批量补录。
2. **智能路由与多模态解析**：
   - 系统根据输入格式（Audio/Image/PDF），自动路由至对应的预处理组件（ASR工具提取文本、OCR/VLM工具提取图文位置及表格）。
3. **大模型意图识别与结构化提取**：
   - AI Agent根据建筑工程领域的特定 Schema（如施工部位、出勤人数、机械使用、安全隐患），对散乱文本和图表进行信息抽取、清洗与纠错。
4. **质量校验与反馈**：
   - 内部 Critic 模型对比提取结果与原始输入，如存在严重“幻觉”或关键字段缺失，则退回补充或标记为低置信度。若识别出高危隐患，将触发内部系统弹窗熔断逻辑，终止后续自动化流转并立即报警。
5. **系统对接与草稿生成**：
   - 将结构化后的标准 JSON 数据，转换为现有手动录入系统的 API Payload。
   - 推送至现有系统，生成处于**“草稿（Draft）”**状态的施工记录单。
6. **人工极速复核**：
   - 录入员在现有系统中打开草稿，核对无误后一键提交审批。

## 4. 详细功能需求

### 4.1 多模态数据接入模块
- **语音输入 (Voice/Audio)**：支持对接前端传来的音频流或录音文件，具备工程领域术语降噪与识别能力（ASR），并在底层追加强效的音频降噪策略以应对复杂环境。
- **图像输入 (Image)**：支持 JPG/PNG 等格式，容忍常见的透视变形、环境光差、部分遮挡及手写体识别。照片强制增加防伪水印（GPS+时间戳校验），并补充印章与签字的检测识别功能。
- **文档输入 (PDF/Excel)**：支持多页PDF扫描件及历史Excel表格的结构化读取与长文本切分。

### 4.2 历史数据批处理调度模块 (Batch Processing)
- **批量任务创建**：支持上传ZIP包或选定文件目录，创建一个“批处理补录任务 (Batch Job)”。
- **并发与限流控制 (Throttling)**：在调用底层大模型API时，必须具备速率限制与拥塞控制功能，防止打爆模型配额。
- **纯附件自动归档降级**：当遇到大量泥污、严重折损导致大模型无法 OCR 解析的极端表单时，放弃强行结构化提取，直接将原始图片作为纯附件归档降级。
- **端侧小模型辅助打标**：为解决死信队列降级带来的无序问题，在设备本地（端侧）部署轻量级视觉小模型，提取图片粗略特征并提供智能推荐标签（如：日期、表单类型），供操作员一键确认。同时，**须定期监控和评估端侧小模型对移动端设备的耗电量、发热量及本地内存占用**，确保满足移动现场作业的稳定性要求。
- **进度监控大屏**：提供可视化的批处理进度看板，实时查看“处理中/成功/失败”统计。
- **死信队列与重试 (DLQ)**：对解析严重失败的记录隔离存放，支持人工修正源文件后一键重发。

### 4.3 核心业务实体化与校验 (Schema Alignment)
本系统生成的“草稿”必须与现有系统施工记录录入所需的字段完全一致。所有智能提取的数据必须强制对齐以下业务表单结构，特别是与前置主数据模块进行精准匹配。

#### 4.3.1 必填主数据字段（柔性主数据对齐）
以下字段属于前置模块已维护好的标准数据，AI 提取文本后必须与系统现有数据库中的标准字典/台账进行**实体解析（Entity Resolution）与对齐匹配**（如模糊匹配、向量检索对齐）：
- 若匹配成功，则映射为系统标准台账数据。
- 若匹配度低于阈值（例如遇到**计划外施工**），则废除过去的直接置空逻辑，转为**柔性主数据对齐策略**：保留原始非结构化文本，自动追加 `[待定项/计划外]` 标识，并且在本地日志中仅作为纯文本记录（不上报新增主数据申请），最终交由人工复核环节顺带补充或修正主数据库。
- **工点**：具体施工地点或标段。
- **队伍**：具体的分包班组或施工队伍名称。
- **分项**：对应的单位/分部/分项工程划分。
- **部位**：具体的空间网格或构件部位。
- **工序**：具体的施工作业动作（如钢筋绑扎、混凝土浇筑等）。

#### 4.3.2 工序明细字段（动态提取与拓展）
基于上述“工序”实体，需进一步提取工序记录的详情数据：
- **工序名称**：标准的工序词汇。
- **工序记录字段（预设可选与动态提取）**：
  - 形象进度类：**今日完成形象量**、**开累完成形象量**、**剩余形象量**。
  - 统计指标类：**今日完成统计量**、**开累完成统计量**、**剩余统计量**。
  - **其他自定义字段**：支持现有系统可手动定义的其他动态扩展字段。

#### 4.3.3 补充管理类字段（大文本与实体混合）
除核心进度外，系统还需支持从繁杂日志或语音中抽取以下维度的管理信息：
- **资源投入**：**作业人员**（出勤明细）、**机械设备**（调配使用情况）。
- **流程与责任**：**工前交班**、**值班领导**。
- **质量与安全（质安环）**：
  - **质量检查验收情况**。
  - **验收存在问题**。
  - **安全检查情况**。
  - **相关部位检查落实整改情况**。
  - *(安全熔断规则：若在安全检查中 AI 识别到如“塌方、渗漏、脚手架变形”等敏感高危隐患词汇，立即触发熔断机制（Bypass Alert），越过系统常规草稿审核箱流转，直接通过**内部系统消息弹窗**即时告警相关负责人！)*
- **兜底信息**：**备注**（提取无法归类的其他杂项说明）。

*(注：凡未能对齐必填主数据的，系统遵循柔性文本保留原则处理，并在前端 UI 预警中高亮标记带有“[待定项/计划外]”的字段，提示人工复核并修正。)*

### 4.4 现有系统适配器 (System Adapter)
- **草稿推送接口集成**：负责将 AI 的 `TaskState` 终态标准数据，转换为现有表单系统兼容的消息格式。
- **外键关联与跳转回溯**：在生成的草稿中，必须保留原始多模态文件的指针链接（如图片 CDN URL、录音播放链接），方便复核人员同屏比对。

## 5. 非功能需求

### 5.1 性能与可扩展性
- **吞吐量**：批处理模式下，支持每日不少于 10,000 页历史档案的处理。
- **状态流转瘦身**：在 LangGraph 等状态机流转中，采用**引用传递（图片、音频存入 OSS/S3，仅传递 URL）**，防止系统内存溢出。

### 5.2 可观测性 (Observability)
- 必须引入全链路追踪（Tracing），记录每一次由于环境嘈杂导致的语音识别错误、OCR乱码、和 LLM 具体输出，以便于后期 Prompt 调优和定责。

### 5.3 数据安全与隔离
- 工程图纸与内部进度包含敏感商业信息。模型交互需确保数据隐私，优先采用私有化部署模型，或企业级脱敏后调用 API。

### 5.4 法定合规与证据固定
工程日志与单据是项目结算及法定索赔的核心凭证，系统必须建立防伪取证链路：
- **照片防伪确权链路**：现场照片拍摄时必须由端侧强制压入不可篡改的“时间戳+GPS坐标定位+防伪水印”，杜绝后期翻拍造假。
- **印章/签字专项防伪模型**：保障业务单据合规，引入针对总包公章、监理章及手写签字签批的视觉实体校验能力，以固定法务维度的证据链。

## 6. 演进路线
- **阶段一：基石与试点**。打通“图片/PDF -> AI结构化 -> 现有系统草稿箱”的核心单据链路，跑通首个历史批处理任务。
- **阶段二：语音提效**。引入 ASR，适配一线人员“边走边说”的移动端快捷录入。
- **阶段三：自我进化**。基于人工修改草稿后的最终定稿数据（Feedback Loop），微调大模型提示词及评估权重，持续降低退回率。
