## 死信队列

### 什么情况任务进入死信队列

在“施工记录智能分析系统”的设计中，任务进入死信队列（DLQ，Dead Letter Queue）或被标记为人工介入挂起状态（NEED_HUMAN_REVIEW / DLQ），主要分为两大类触发场景：

1. 重试耗尽（系统与网络级异常）
这类情况通常是由暂时性、可恢复的基础设施问题引起的。系统会先尝试抢救，但在达到最大重试次数后彻底放弃，将其放入死信队列。

网络瞬断与超时：调用外部服务（如 vLLM 大模型接口、遗留系统 REST API）时发生 ReadTimeout 或 ConnectionError。
资源被持续挤兑：连续多次获取不到全局限流的令牌（Token Bucket 被掏空），并在退避缓冲队列（q_dlq_retry）中停留超过了既定阈值。
重试策略机制：系统配置了指数退避调度（如 1s -> 2s -> 4s -> 8s）。当一个任务在重试队列中连续失败达到了 max_retry（例如 3 次或 4 次）后，Celery 调度器会放弃抛库，触发 dlq.py 中的死信处理钩子，硬截断进死信队列。
2. 快速失败 - Fast-Fail（数据与业务级致命异常）
这类情况通常是由于输入数据本身或业务逻辑发生了不可逆的错误。系统判断“再重试多少次都没用”，因此跳过重试等待周期，直接将其打入死信队列（通常库中状态标记为 NEED_HUMAN_REVIEW）。

具体表现为以下几种情况：

数据源破损与非法：前端传来的文件经过预处理模块（preprocess）时，发现文件损坏无法拆页、魔数（File Magic Number）不匹配、防伪校验彻底不通过。
业务特征缺失（全白页/无价值数据）：经过视觉或音频初步检查，发现图像是全白的、或者录音全是环境底噪没有任何人声，直接作废挂起。
AI 反思引擎陷入死胡同（连续同错）：
在 Plan-Execute-Reflect 循环中，大模型提取的数据死活无法通过硬规则校验（例如：“浇筑方量”与“总价”在逻辑上存在极其严重的矛盾）。如果大模型尝试调用截图工具、重新提取等动作 ≥3次 依然无法满足金额平衡或日期校验规则，系统判定 AI 无法自动解决该复杂图纸，直接推入死信。
四级对齐彻底失效：在强制向遗留系统对齐（aligner.py）时，如果在 Level 4 阶段（向量库泛化检索也无法映射到旧系统现存的外键 ID），则该任务作为 Unmapped 报废，推入死信队列等待人工手动映射字典。

### 死信队列的任务什么时候清除

1. 业务干预清除：人工纠偏与审核完成（主路径）
这是业务上最常见的清理方式。正如 Detailed_Design.md 中设计的 人机协同模块：

当任务彻底失败进入死信队列时，其在数据库（PostgreSQL）中的状态会同步变为 NEED_HUMAN_REVIEW 或 DLQ。
审核员在中后台的“三分屏”界面调用 PUT /api/v1/tasks/{task_id}/correction 提交了人工修正后的数据。
接收到人工提交后，系统会将数据落库，向遗留系统推送，同时向消息队列发送一条 ACK（确认消费） 指令，或者执行一个出队操作，将该任务从 q_dlq_retry 中永久清除。
2. 运维干预清除：批量重放（Batch Replay）
很多时候任务进入死信是因为下游系统（如：旧版 ERP）宕机或网络中断。

当故障恢复后，系统管理员会调用运维接口（如 POST /api/v1/admin/tasks/requeue）。
这个接口会遍历死信队列，将里面的任务**重新打包并移动（Move）**回 q_batch 或 q_realtime 队列去重新执行。
一旦任务被成功转移回主队列，它在死信队列中的原有消息就会被清除。
3. 定时调度清除：Celery Beat 归档清理（兜底流转）
即便有人工审核，也难免会有部分任务长期无人处理被“遗忘”。此时就需要借助定时任务来打扫战场（对应我们源码结构中的 src/worker/tasks/schedule.py）。

清理策略：配置一个 Celery Beat 定时检查任务（例如每天凌晨 2 点执行）。
它可以设定一个业务阈值，比如“超过 7 天未被处理的 DLQ 任务”。
定时任务会拉取这些过期死信，确认其关键错误日志已经妥善安全地保存在了关系型数据库（ai_parse_tasks.error_message）中，然后判定该任务属于“作废/丢弃（ABORTED）”终态。
随后，定时任务主动向队列发出清除指令，将占用的 Redis 内存释放。
4. 物理防线清除：消息 TTL (Time-To-Live) 过期
为了防止由于系统 Bug 导致定时清理脚本挂掉、进而拉爆 Redis 内存，我们会在基础设施层面设置最后一道防线：

在向 q_dlq_retry 写入消息时，会附带一个全局的 TTL 属性（例如 14 天或 30 天）。
一旦达到这个物理时间上限，RabbitMQ 或 Redis 会在底层直接丢弃该消息。
安全性：由于进入死信的那一刻，数据库中已经留下了这笔任务作为 FAILED 状态的完整快照，因此即便消息队列里的运行上下文因 TTL 物理过期被删，我们依然不会丢失这笔“账”，依然可以在数据库里查到这次失败的记录。

总结
  死信队列的清理是一个**“从上到下、由重到轻”**的漏斗：
  优先被前端人工修补“救活”并清除；
  其次被运维人员批量重放移走；
  然后被定时脚本 (schedule.py) 作为过期废件扫地出门；
  最后由中间件的 TTL 过期机制 物理销毁。

通过这套机制，既保证了业务数据不因程序异常而丢失，又保证了昂贵的 Redis/内存资源不会被无限拖垮。

### 为什么要有死信队列，直接存入数据库不行吗

如果“直接把失败任务状态在数据库里 update 为 failed，然后搞个定时任务去扫表重试”，虽然看似简单，但在**高并发、大吞吐（日处理 ≥ 10,000 页多模态文档，特别是依赖昂贵且极其脆弱的 GPU 大模型资源）**的场景下，会引发严重的灾难。

引入死信队列（DLQ）机制而不是仅仅依赖数据库状态更新，主要基于以下五个核心原因：

1. 保护数据库，避免“慢查询拉爆连接池” (Thundering Herd Problem)
如果我们只用数据库：
系统为了实现重试或错误恢复，必须写一个守护进程（Cron / Polling）。这个进程需要高频地执行：SELECT * FROM tasks WHERE status = 'FAILED' AND next_retry_time <= NOW()。
在正常情况下这还好，但一旦底层遗留系统网络波动，或者 GPU 显存 OOM 导致大面积抛错，表中瞬间堆积数万条错误任务。几十个 Worker 疯狂轮询数据库，不仅造成巨大的 I/O 和锁竞争，还极易拉爆 PgBouncer 的连接池，导致正常的前端写入请求（API 进件）也被拖垮。

有了死信队列：
所有需要等待、退避、失败挂起的任务都在基于内存分布式的 Broker（Redis/RabbitMQ）中流转。出借、等待、回推的操作对于 Redis 而言通常仅需不到 1 毫秒且单线程原生存放。数据库退化成了纯粹的“快照留痕”存储，而不是“调度驱动器”。

2. 完美的“指数退避与毒药隔离” (Exponential Backoff & Poison Pill Isolation)
**毒药消息（Poison Pill）**是指那些因为某种畸形的数据结构，导致任何读取它的进程都会瞬间崩溃退出的任务。
如果仅仅保存在数据库并通过轮询取回，这颗“毒药”会被一遍遍地拉出来，一遍遍地搞崩你的工作进程。

死信队列原生附带了完善的退避控制环。以 Celery 的运行机制为例，当异常发生推入 q_dlq_retry 时，调度器可以在消息头打上标签（如：{retry_count: 2, wait_for: 8s}）。如果达到最大次数，它会彻底安息在死信槽，绝对不会再被主干算力流（q_realtime / q_batch）触碰到。它完美实现了“毒药隔离”。

3. 解救昂贵的 GPU 算力通道占用
在 AI 解析引擎设计中，显存是极其宝贵的（比如系统中我们限制了 GPU Inflight 为 20）。
如果失败任务和实时大吞吐任务混杂在相同的逻辑通道处理，失败重试产生的阻塞很容易占满执行槽位（即所谓的 Head-of-line blocking，队头阻塞）。

引入专门的死信消费者：
将多次失败的任务丢入独立的 DLQ，我们可以：

异步排空：主进程立刻释放它对网络和显存的上下文占用。
错峰填谷：你可以专门启动一台很廉价的低配机器（或限制消费速率权重极低的 Worker）去缓缓消费这个死信队列。遇到故障我们不急着去死磕，而是让出主赛道，保障健康的任务能够丝滑全速前进。
4. 保障系统容灾时的“无缝回放与现场恢复” (Replayability)
将死信视为“未完成的事件体（Event Payload）”比将其视为“数据库里的一条死记录”更有实操价值。

死信队列里存放的是触发这个动作时最原始、最完整的消息上下文（包括请求载荷、TraceID、上一步的快照、异常堆栈序列）。
当系统崩溃（比如下游那个老旧的基建 ERP 系统宕机维护了 24 小时）恢复后，修复人员只需要触发一条指令，将 DLQ 里的几十万条堆积消息原封不动地重新放回 q_batch 或 q_realtime 中（Replay）。
如果纯靠读数据库还原，你必须编写大量“如何把数据库字段重新序列化组装成任务上下文”的补偿代码，不仅极易出错，而且会导致严重的业务逻辑耦合。

5. 更好的可观测性接入与告警阻抗
在监控体系（Prometheus / Grafana）中，MQ/Redis 的队列深度（Queue Depth）是一个极其敏锐且成熟的探针。
死信队列（DLQ）的长度只要 >0，或者增长率突然变陡，监控面板立马可以触发 P0 级电话告警。运维在收到告警时，用 MQ 可视化工具（如 Flower）看一眼死信队列，马上就能直接抓取到导致集体阵亡的那截网络超时堆栈进行诊断。如果单纯扫关系型库拉表，指标统计既有滞后性也很耗计算资源。

总结
在这个多模态架构中：
  PostgreSQL (ai_parse_tasks) 负责**“What（发生了什么）：审计、前端展示、持久化归档”**。
  死信队列 (q_dlq_retry) 负责**“How/When（怎么流转、何时挂起）：时机控制、算力隔离、安全重放”**。
这种“状态存库，控制在队列”的职责分离，正是支撑日活万级高并发与保证 GPU 昂贵资源不被阻塞的核心。
